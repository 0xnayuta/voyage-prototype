# ADR-0001: Server Action 作为游戏操作权威入口

---

## 背景

游戏操作（买入、卖出、航行等）需要一个权威的执行路径。单机离线游戏状态下，客户端和服务端在同一进程中，有两个可能的执行模型：

1. **客户端先算**：浏览器先执行游戏逻辑，更新 UI，再异步保存到 SQLite
2. **Server Action 权威**：所有操作提交到 Server Action，执行逻辑、保存 SQLite、返回最新 World、刷新 UI

---

## 备选方案

### 方案 A：客户端先算 + 异步保存

```
用户点击 → 客户端执行逻辑 → 更新 UI → 异步保存到 SQLite
```

优点：
- 无延迟感，UI 即时响应

缺点：
- UI 状态与存档状态可能不一致（异步保存失败、浏览器崩溃、竞态条件）
- 游戏逻辑散落在客户端和服务器两端，需要两套校验
- 调试时需要同时检查客户端 store 和 SQLite，对不上的情况难以定位
- 增加新字段需要在 store 类型 + 序列化逻辑 + SQLite 三处修改

### 方案 B：Server Action 权威

```
用户点击 → Server Action → 执行逻辑 → 保存 SQLite → 返回 World → 刷新 UI
```

优点：
- World = 唯一真相源，不存在同步问题
- 所有游戏逻辑集中在 `src/game/` 纯函数中，与 UI 框架解耦
- 存档是每次操作的天然副产品，不存在「忘记存档」
- 调试只需查 SQLite
- 浏览器崩溃不丢失进度（SQLite 中始终是上一个完整状态）

缺点：
- 每次操作有网络往返延迟（100-200ms）

### 方案 C：纯客户端 + 本地存储

什么都不用，直接在客户端用 IndexedDB/localStorage 操作。

优点：没有往返延迟

缺点：
- 游戏逻辑和 UI 耦合
- 无法享受 TypeScript 类型安全和 Prisma schema 管理
- 存档恢复逻辑弱
- 未来 Electron 迁移需要重写数据层

---

## 决策

**采纳方案 B：Server Action 权威。**

### 理由

该游戏是**回合制策略游戏**，不是即时反应游戏：

- 每步操作之间用户天然在阅读/思考/选择
- 100-200ms 往返在可接受范围内（禁用按钮 + "处理中..." 即可）
- 方案 A 牺牲了数据一致性换取「即时响应」——对这个游戏类型来说是不必要的

核心权衡判断：

| 维度 | 方案 A | 方案 B（采纳） |
|---|---|---|
| 状态一致性 | ❌ 概率性不一致 | ✅ **不可能不一致** |
| 崩溃容错 | ❌ 可能丢当前操作 | ✅ 前一个完整状态 |
| 调试难度 | 难（两处状态对不上） | 易（查 SQLite 即可） |
| 延迟 | 无 | 100-200ms（策略游戏可接受） |
| 架构复杂度 | 高（同步协调逻辑） | 低（线性数据流） |

---

## 后果

### 正面

- UI 组件只需等待并渲染 Server Action 返回的 GameView，不需要维护本地游戏状态副本
- 游戏中所有「预览」类数据（预估利润、航行天数）由 View Builder 在 Server Action 内计算，不会出现「客户端看到的和服务器算的不一样」
- 加入 Electron 壳后，架构无需调整——Server Action 依然在同一进程中运行

### 约束

- 组件必须处理 `isPending` 状态，禁用按钮防重复提交
- 不可在浏览器端对 World 做写操作——所有 `useState` 只保存 UI 临时状态

### 需要注意

- 快速连点虽然被 `disabled={isPending}` 封住了 95% 的情况，但极端场景（如第一次 Action 刚完成 UI 未刷新时第二次点击）仍可能触发并发。MVP 阶段不处理，Phase 2 引入后台任务时加 Version 乐观锁
